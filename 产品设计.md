# 一、产品功能边界（MVP 锁定）

**用户角色**：教师（核心）、家长、机构管理员
**学科**：K12 数学（单学科先跑通）
**输入**：图片（多张）/ PDF，含三种情况：
  - A. 纯空白试卷（用于建题库）
  - B. 试卷 + 答题卡（分离）
  - C. 试卷上直接作答（含手写）

**核心功能**：
1. 试卷 OCR + 结构化拆题
2. 自动批改（客观题确定性判分 + 主观题 LLM 评分 + 教师审核）
3. 错题归因（错误类型 + 知识点定位）
4. 知识点图谱 + 薄弱点雷达图
5. 学习计划生成
6. 班级/年级横向对比
7. 学生历史趋势分析

# 三、关键数据模型（精简版）

**users**：id, role(teacher/parent/admin), org_id, name, phone, …
**organizations**：机构
**classes**：班级（属于机构）
**students**：id, class_id, parent_user_id, grade, …
**knowledge_points**（树形）：id, parent_id, name, grade, subject, level, description
**papers**：试卷元数据（id, title, grade, source, owner_id, created_at）
**questions**：题目（id, paper_id, number, type, stem, options, answer, analysis, knowledge_point_ids[], difficulty）
**answer_sheets**：一次作答（id, paper_id, student_id, raw_files[], status）
**student_answers**：学生答题（id, sheet_id, question_id, raw_text, parsed_answer, is_correct, score, ai_score, teacher_score, error_type, error_reason, reviewed_by, reviewed_at）
**student_kp_mastery**：学生-知识点掌握度（student_id, kp_id, mastery_score, last_updated, sample_count）
**study_plans**：学习计划
**audit_logs**：教师审核日志

# 四、核心业务流程

## 4.1 试卷处理流水线（最重要）

```
上传(图片/PDF)
  │
  ▼
[1] 预处理：去重、纠偏、切页、分辨率归一化
  │
  ▼
[2] 版式识别 (VLM)：判断是 空白试卷 / 试卷+答题卡 / 试卷作答
  │
  ▼
[3] 题目结构化抽取 (VLM)
   - 输出 JSON：题号、题型、题干、选项、图片裁剪框
   - 数学公式输出 LaTeX
  │
  ▼
[4] 学生作答抽取 (VLM)  ← 仅 B/C 情况
   - 选择题：识别涂卡/勾选
   - 填空/计算：识别手写文本+公式
  │
  ▼
[5] 知识点对齐 (LLM + 向量召回)
   - 用题干向量去 KP 库召回 Top-K，再让 LLM 二次确认
  │
  ▼
[6] 批改
   - 客观题：规则判分
   - 主观题：LLM 按"参考答案 + 评分细则"打分，输出步骤分
  │
  ▼
[7] 错题归因 (LLM)
   - 错误类型分类(概念性/计算性/审题性/方法性/书写性)
   - 错误原因解释 + 正解思路
  │
  ▼
[8] 教师审核 (人工)
   - 列表式审核界面，AI 结果可一键采纳/修改
  │
  ▼
[9] 写入画像
   - 更新 student_kp_mastery（贝叶斯/EWMA 更新掌握度）
```

> **关键设计**：第 3、4 步要解耦——空白试卷只跑一次"题目结构化"入题库；后续学生作答直接复用题目结构，只跑作答识别，**大幅节省 token**。

## 4.2 知识点掌握度更新（建议算法）

对每个知识点采用 **指数加权移动平均（EWMA）**：

```
new_mastery = α * current_correct_rate + (1-α) * old_mastery
α 随样本量自适应，样本越多 α 越小（越稳定）
```

再结合 **难度加权**：做对难题加分多，做错简单题减分多。

## 4.3 学习计划生成

输入：学生最近 N 次画像 + 薄弱 KP Top-K + 年级目标
LLM 输出结构化 JSON：
```json
{
  "周计划": [
    {"日期":"周一","知识点":"一元二次方程","任务":[...],"预计时长":30}
  ],
  "推荐题目": [question_id,...],
  "重点提醒": "..."
}
```

# 五、AI 调度层设计（重点）

这是你**多模型 API 并存**的关键，建议做一个 **Model Router**：

```python
class ModelRouter:
    def call_vlm(self, task_type, payload, quality="standard"):
        # task_type: layout / question_extract / answer_extract
        # 不同任务路由到不同模型，例如：
        # - 版式识别：用便宜的国产 VLM
        # - 题目结构化（含公式）：用 GPT-4o / Qwen-VL-Max
        # - 学生手写识别：用对手写最强的模型
        ...
    def call_llm(self, task_type, payload):
        # 归因/计划：用强推理模型
        # 简单分类：用小模型
        ...
```

**节省 token 的关键策略**：
1. **图片只在必要环节传给 VLM**，归因/计划等纯文本任务交给 LLM。
2. **题目结构化的结果落库复用**（同一份试卷只抽一次）。
3. **批量处理**：一张图里多道题一次性抽，而不是一题一次。
4. **结构化输出强约束**：用 JSON Schema / function calling，避免重复重试。
5. **缓存**：相同图片哈希命中直接返回结果。
6. **分级模型**：先用小模型粗筛，疑难再升级到大模型。

# 六、Prompt 设计要点（给你几个核心模板）

### 6.1 题目结构化（VLM）

```
你是数学试卷解析助手。请从图片中提取所有题目，输出 JSON 数组。
要求：
- 数学公式使用 LaTeX，包裹在 $...$ 内
- 图形/图表用 [图: 描述] 占位并给出 bbox
- 字段：number, type(choice/fill/calc/proof/solution), stem, options, sub_questions
- 严格输出 JSON，不要解释
```

### 6.2 错题归因（LLM）

```
已知：
- 题目：{stem}
- 标准答案：{answer}
- 解析：{analysis}
- 学生作答：{student_answer}
- 关联知识点：{kp_list}

请输出 JSON：
{
  "is_correct": bool,
  "score": 0-100,
  "error_type": "概念性|计算性|审题性|方法性|书写性|无",
  "error_step": "学生第几步开始出错",
  "root_cause": "一句话归因，指向具体知识点掌握缺陷",
  "suggested_kp_to_review": [kp_id,...],
  "mini_tutorial": "≤80 字针对性提示"
}
```

# 七、前端设计

## 7.1 技术选型
- **框架**：React + TypeScript + Vite
- **UI**：Ant Design 5（教育/管理后台风格契合度高）
- **状态**：Zustand / Redux Toolkit
- **图表**：ECharts（雷达图、趋势图、知识图谱）
- **公式渲染**：KaTeX
- **PDF/图片预览**：pdf.js + 自研裁剪组件
- **任务进度**：WebSocket / SSE

## 7.2 主要页面（按角色）

### 教师端
1. **首页 Dashboard**：班级概览、近期试卷、待审核数量
2. **试卷管理**
   - 上传向导（选择"建题库"或"批改作答"）
   - 试卷列表 / 详情
3. **批改与审核**（核心页面）
   - 左侧原图 + 框选高亮
   - 右侧题目结构 + AI 批改结果 + 编辑表单
   - 一键采纳 / 批量通过
4. **班级分析**
   - 班级雷达图（知识点掌握度均值）
   - 题目正确率排行
   - 学生横向对比表
5. **学生详情**
   - 个人雷达图、趋势曲线、错题本、学习计划
6. **题库 & 知识点体系**
   - 树形知识点维护
   - 题目 CRUD（含富文本 + LaTeX）

### 家长端
1. 孩子画像首页（薄弱点、近期表现）
2. 错题本（按知识点筛选）
3. 学习计划查看
4. 历史趋势

### 机构管理端
1. 班级/教师/学生管理
2. 模型用量与费用统计
3. 跨班级横向对比

## 7.3 关键交互设计点
- **审核页**必须支持 **图片区域 ↔ 题目结构** 的双向高亮联动，否则教师效率很低。
- **上传流程**用向导式（Stepper），让用户先选场景（建题库 / 批改）。
- **长任务**前端显示分阶段进度（OCR → 识别 → 批改 → 归因），减少焦虑。

# 八、后端技术选型推荐

| 模块 | 选型 | 理由 |
|---|---|---|
| Web 框架 | **FastAPI** | 异步、文档自动化、AI 生态匹配 |
| 任务队列 | **Celery + Redis** | 成熟，便于水平扩展 |
| 数据库 | **PostgreSQL** | JSONB 字段方便存题目/作答 |
| 向量库 | **PGVector** | MVP 阶段无需引入 Milvus |
| 对象存储 | **MinIO** (自建) 或 **OSS** | 存图片/PDF |
| 缓存 | **Redis** | 通用 |
| 部署 | **Docker Compose**（MVP） → K8s | 200-500 并发先单机+横向扩 worker |
| 监控 | Prometheus + Grafana + Sentry | 必备 |

**并发评估**：200-500 并发不算高，瓶颈在 **VLM API 调用延迟**。建议：
- 上传是同步、识别批改是**异步任务**
- Worker 池单独扩容，按模型 QPS 限速
- 同一份试卷的多张图**并行调用** VLM，再汇总
