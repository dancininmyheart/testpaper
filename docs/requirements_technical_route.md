# 试卷智能分析平台需求与技术路线文档

## 1. 文档目的

本文档基于当前仓库已有实现、README、二次重构设计文档和 demo 分析链路，总结项目的产品需求、业务边界、技术现状与后续技术路线。

文档目标：

- 明确当前项目要解决的核心教学问题。
- 固化 v1 的交付范围，避免早期范围发散。
- 把已验证的 demo 能力沉淀为平台化技术路线。
- 为后续开发、测试、重构和验收提供统一依据。

## 2. 项目定位

当前项目是一个面向教师和教学管理场景的试卷智能分析平台。系统通过上传试卷、答题卡和标准答案，自动完成题目识别、学生作答识别、标准答案判定、错因诊断、报告生成，并将结果沉淀为学生知识点掌握度记录。

一句话定位：

```text
把“试卷批改 -> 错因分析 -> 学生画像 -> 掌握度追踪”做成可复用、可审计、可扩展的教学分析闭环。
```

## 3. 当前业务现状

### 3.1 已验证能力

当前仓库已经验证了完整 demo 闭环：

- 支持上传试卷、答题卡、标准答案或自动生成参考答案。
- 支持图片和 PDF 输入，服务端可将 PDF 转为可分析页面。
- 使用 VLM/LLM 识别题目结构、学生作答、标准答案和错因。
- 结合标准答案与教师批改痕迹生成最终判定。
- 输出 `answer_trace`、`reference_answers`、`student_profile`、`analysis_process.stages` 等结构化结果。
- 支持 JSON/PDF 报告导出。
- 已有掌握度 MVP，可将结构化诊断记录转化为学生知识点掌握度。
- 已有平台化后端，包含用户、会话、分析任务、项目、文件持久化、审计日志和查询接口。

### 3.2 主要问题

当前核心问题不是能力缺失，而是工程边界不够清晰：

- `demo/service.py` 承担了输入校验、文件标准化、模型调用、题目识别、答案识别、对齐、判分、画像、报告导出等多重职责，文件过长，维护成本高。
- demo 输入模式较多，主流程分支复杂，容易让修复逻辑扩散。
- 上传标准答案和自动生成标准答案在部分链路中容易混用，结果可信度需要更清晰表达。
- 模型配置、运行时 profile、密钥和调试开关仍较多依赖本地 JSON 与环境变量，需要进一步治理。
- 缺少统一评测集和量化指标，难以稳定判断题目识别、答案映射、判分质量是否变好。
- 平台化 `backend/` 已经出现，但核心分析能力仍主要桥接 legacy demo，后续应逐步抽离阶段。

## 4. 用户与场景

### 4.1 目标用户

- 教师：上传试卷和答题卡，查看批改结果、错因分析、学生报告。
- 教研人员：查看班级或试卷层面的知识点薄弱项和题目表现。
- 管理员：管理账号、配置模型、查看任务状态和系统运行情况。
- 开发与测试人员：通过 mock、样例数据和阶段日志验证链路质量。

### 4.2 核心场景

#### 场景 A：单学生单次试卷分析

教师上传一份试卷、一份学生答题卡和一份标准答案，系统异步完成分析，返回结果 JSON 和 PDF 报告。

这是 v1 的主链路。

#### 场景 B：试卷项目复用

教师先创建试卷项目，上传试卷和标准答案，系统提取题目与参考答案；教师审核通过后，后续多个学生答题卡可以复用同一套题目结构和标准答案。

这是当前平台化代码中已经出现的方向，但 v1 可先保证单学生闭环稳定，再逐步强化项目复用。

#### 场景 C：掌握度追踪

系统将判分和错因诊断转为掌握度事件，更新学生在不同知识点上的掌握度、近期表现和薄弱项。

## 5. v1 需求范围

### 5.1 v1 必做

v1 聚焦最稳定、最容易验收的路径：

```text
上传试卷 + 上传答题卡 + 上传标准答案
-> 文件标准化
-> 题目结构识别
-> 答题区域识别
-> 学生作答识别
-> 标准答案解析
-> 题目/作答/标准答案对齐
-> 判分与错因分析
-> 生成报告
-> 掌握度事件入库
```

必须交付：

- 登录、会话和教师/管理员基础权限。
- 创建分析任务、查询任务列表、查询任务详情。
- 上传试卷、答题卡、标准答案。
- 后台异步执行分析任务。
- 任务失败记录错误信息，支持失败重试。
- 分析结果 JSON 查询。
- 分析报告 JSON/PDF 下载。
- 阶段级日志可见，至少包含阶段名、状态、耗时和错误。
- 掌握度事件写入和学生掌握度查询。
- mock 与真实分析链路隔离，避免误把演示结果当真实结果。

### 5.2 v1 暂不做

以下能力后置，避免初期再次复杂化：

- 自动生成标准答案作为主流程能力。
- 同页作答模式。
- 批量学生批改的完整体验。
- 模板任务复用。
- 多模型动态对比和自动择优。
- 人工框选答案块的完整工作流。
- 复杂多轮修复策略。
- 知识点图谱自动写回。

这些能力可以保留接口扩展空间，但不进入 v1 的主成功标准。

## 6. 功能需求

### 6.1 用户与权限

- 支持教师和管理员角色。
- 支持登录、登出、会话过期。
- 教师只能查看自己创建的任务和项目。
- 管理员可查看全量任务。
- 关键操作写入审计日志，例如创建任务、重试任务、导入掌握度事件。

### 6.2 文件上传与持久化

- 支持上传图片和 PDF。
- 文件按任务或项目落地保存。
- 分析任务只保存文件引用、元数据和 payload，不把大文件直接塞入数据库。
- 记录文件分类：`paper_files`、`answer_sheet_files`、`answer_key_files`、`combined_files`。
- 保存原始文件名、content type、大小和本地路径。

### 6.3 分析任务

- 创建任务后进入 `queued`。
- Worker 拉取队列任务并置为 `running`。
- 成功后置为 `succeeded`，保存结果和阶段日志。
- 失败后置为 `failed`，保存错误信息。
- 支持 failed/canceled 任务重试。
- 任务状态建议统一为：

```text
queued -> running -> succeeded
              |-> failed
              |-> canceled
```

### 6.4 试卷项目

- 支持创建试卷项目，保存标题、学科、年级、创建人和状态。
- 支持上传项目级试卷和标准答案。
- 支持触发题目和标准答案提取。
- 支持教师审核题目结构，审核后项目进入 ready 状态。
- 支持基于 ready 项目分析学生答题卡。
- 支持保存评分复核数据和最终报告。

项目状态建议：

```text
draft -> extracting -> review_questions -> ready
ready -> analyzing -> review_scores -> completed
任意关键阶段 -> error
```

### 6.5 题目结构识别

- 从试卷页面中提取题号、题型、题干、小题、分值、页码范围和候选知识点。
- 输出结构应能支持后续 answer mapping 和判分。
- 低置信度或缺失字段不应静默丢弃，应在阶段日志或结果中体现 warnings。

### 6.6 答题区域与学生作答识别

- 从答题卡中识别学生作答区域。
- 客观题识别选项。
- 填空题识别文本。
- 主观题识别步骤、最终答案、教师批改痕迹和分数线索。
- 输出应能回溯来源页、区域、置信度和原始模型响应。

### 6.7 标准答案解析

- v1 以上传标准答案为主。
- 标准答案输出需包含题号、答案文本、最终答案、步骤、来源和置信度。
- 如果标准答案缺失，应明确标注缺失题目，不能伪装成已判定。
- 自动生成标准答案可以作为后续扩展或 demo 兼容能力，但不作为 v1 主验收路径。

### 6.8 对齐、判分与错因分析

- 建立 `Question <-> StudentAnswer <-> ReferenceAnswer` 的稳定关联。
- 优先使用标准答案进行判定。
- 教师批改痕迹作为辅助信号，可用于冲突提示。
- 输出每题最终对错、置信度、原因、冲突信息和错因。
- 对低置信度、缺失答案、标准答案缺失、题号不匹配等情况生成可解释状态。

### 6.9 报告与导出

- 支持查询分析结果 JSON。
- 支持下载 JSON 报告。
- 支持下载 PDF 报告。
- 报告应服务教师复核，优先展示题目、学生作答、标准答案、判定、错因和建议。

### 6.10 掌握度闭环

- 将分析结果转换为统一掌握度事件。
- 事件至少包含 `student_id`、`timestamp`、`problem_id`、`source_type`。
- 推荐包含题目文本、学生答案、得分、满分、试卷 ID、知识点 ID 和 LLM 结构化诊断。
- 使用掌握度引擎更新 `student_mastery` 和 `evidence_chain`。
- 支持学生掌握度查询、学生报告、试卷分析和班级知识点汇总。

## 7. 非功能需求

### 7.1 可维护性

- 核心分析能力按阶段拆分，避免单个服务继续膨胀。
- API 层不写业务流程，只做请求解析、认证和响应。
- 应用层负责任务编排和用例服务。
- 领域层定义稳定模型和规则，不依赖 Flask、SQLite 或具体模型供应商。
- 基础设施层封装数据库、文件存储、AI 客户端、PDF/图片处理和报告渲染。

### 7.2 稳定性

- 分析任务必须有明确状态和错误。
- LLM/VLM 调用失败时应按阶段降级，不应让错误来源丢失。
- 关键阶段结果应可追踪，包括输入摘要、输出摘要、耗时和 warnings。
- mock 与真实链路必须隔离。

### 7.3 性能

- LLM-heavy 阶段是主要耗时来源，优化重点应放在分块、并发、按需执行和减少上下文。
- 上传标准答案路径应避免触发自动参考答案生成。
- 自动生成参考答案时，应按题目分块并发，失败题目单独重试。
- 作答结构化应按页、raw item 或题号范围分块，不要一次携带全量上下文。
- 错因诊断和画像应按需执行，优先处理错误、扣分、冲突和低置信题。

### 7.4 可观测性

- 每个阶段记录 `stage`、`status`、`elapsed_ms`。
- 并行阶段建议记录 `started_at`、`finished_at`、`actual_elapsed_ms`、`wait_elapsed_ms`。
- 关键计数进入 `mapping_report` 或阶段日志，例如题目数、答案块数、参考答案生成 chunk 数、跳过题目数、诊断题目数。
- 调试 JSON 可按任务落盘，但应受 profile 或配置开关控制。

### 7.5 安全与配置

- 密钥不应写入仓库提交。
- `.env` 用于本地配置，生产配置应通过环境变量注入。
- `llm_config.json` 可保留 profile 定义，但密钥和敏感项需要逐步迁移到环境变量或密钥管理。
- 文件上传需限制类型、大小和路径穿越风险。
- 用户权限和任务访问控制必须在 API 层统一校验。

## 8. 技术路线

### 8.1 总体架构

当前推荐采用 Flask Fullstack + React 前端 + SQLite 本地持久化 + 后台 worker + VLM/LLM 适配器的轻量平台路线。

```text
frontend
  React + Vite + Ant Design + React Query

backend
  api             HTTP 路由、认证、请求响应
  application     用例服务、任务编排、工作流
  domain          领域对象、状态、端口协议
  infrastructure  SQLite、文件存储、AI 适配器、报告导出
  workers         队列轮询和任务执行

legacy demo
  demo/service.py 当前完整分析能力来源
  demo/stages     逐步抽离后的阶段模块
```

### 8.2 后端技术路线

后端继续保留当前分层，并逐步削弱 legacy demo 的中心地位。

第一阶段：稳定平台壳。

- 保留 `backend/api/routers/*` 作为统一 API 入口。
- 保留 `AnalysisService` 管理任务生命周期。
- 保留 `PaperService` 管理试卷项目生命周期。
- 保留 `MasteryService` 对接掌握度引擎。
- Worker 继续轮询 queued 任务并写回状态。

第二阶段：抽离分析阶段。

- 以 `AnalysisWorkflow` 作为主编排入口。
- 以 `PipelineContext` 承载阶段共享状态。
- 每次只从 `demo/service.py` 抽离一个稳定阶段。
- 已抽离阶段通过 `PipelineOrchestrator` 执行，未抽离部分继续走 legacy adapter。
- 保证每次抽离后结果 JSON shape 不破坏前端和测试。

第三阶段：领域模型固化。

建议逐步定义并稳定以下领域对象：

- `AnalysisJob`
- `PaperProject`
- `DocumentPage`
- `Question`
- `AnswerBlock`
- `StudentAnswer`
- `ReferenceAnswer`
- `Judgement`
- `AlignmentReport`
- `MasteryEvent`
- `AnalysisStageLog`

第四阶段：替换 legacy。

- 当题目识别、标准答案解析、答题识别、对齐判分、错因诊断、报告生成均可作为阶段运行后，`demo/service.py` 退化为兼容入口或测试基线。
- 新链路成为平台默认链路。

### 8.3 前端技术路线

当前前端使用 React、Vite、Ant Design、React Query、Axios 和 Zustand，适合继续承载平台化管理界面。

页面重点：

- 登录页。
- 仪表盘。
- 任务列表与任务详情。
- 试卷项目列表与项目详情。
- 报告中心。
- 学生掌握度页。

前端应优先支持教师的高频工作流：

- 创建项目。
- 上传试卷和标准答案。
- 审核题目。
- 上传学生答题卡。
- 查看任务进度。
- 复核结果。
- 下载报告。

### 8.4 数据存储路线

v1 继续使用 SQLite，满足本地演示、单机部署和快速迭代。

已有核心表：

- `users`
- `sessions`
- `analysis_jobs`
- `analysis_job_files`
- `analysis_results`
- `mastery_events`
- `audit_logs`
- `paper_projects`
- `paper_project_files`
- `paper_questions`
- `paper_reference_answers`

后续可补充：

- `analysis_stage_logs`：把阶段日志从 JSON 字段升级为可查询表。
- `document_pages`：保存文件标准化后的页面资源。
- `student_answers`：保存作答识别结果。
- `judgements`：保存每题判定。
- `report_exports`：保存报告导出记录。

### 8.5 AI 能力路线

AI 能力应以端口和适配器形式隔离：

- VLM：页面理解、题目识别、答题区域识别、学生作答识别。
- LLM：标准答案解析、对齐判定、错因诊断、学生画像、学习建议。
- YOLO/视觉检测：答题区域或版面切分。
- PDF/图片工具：PDF 转图、图片裁剪、页面标准化。

建议策略：

- 每个模型调用都明确输入、输出 schema 和失败处理。
- 将 profile 作为运行策略，不让业务逻辑直接依赖具体模型名称。
- fast profile 用于平衡提速，默认保留结果结构。
- 重要输出使用结构化 JSON schema，失败时保留原始响应和错误。

## 9. 分析流水线设计

v1 主流水线建议如下：

```text
01_validate_input
02_normalize_files
03_extract_questions
04_detect_answer_blocks
05_recognize_student_answers
06_parse_reference_answers
07_align_answers
08_judge_correctness
09_analyze_mistakes
10_build_student_profile
11_ingest_mastery_events
12_build_report
```

每个阶段遵循统一契约：

```text
input: PipelineContext + 阶段所需字段
output: StageResult(status, output, warnings, elapsed_ms)
side effects: 必要时写入文件、数据库或 stage_logs
failure: 返回 failed 或抛出可归因错误
```

阶段拆分原则：

- 一个阶段只做一件事，符合职责单一。
- 阶段之间通过结构化对象传递，减少隐式依赖。
- 阶段失败要能定位，不让用户只看到“分析失败”。
- 可替换能力通过端口注入，符合面向抽象。
- legacy fallback 只作为迁移手段，不能成为新功能默认依赖。

## 10. API 路线

### 10.1 分析任务 API

当前已有接口方向：

- `POST /api/v1/analysis/jobs`：创建分析任务。
- `GET /api/v1/analysis/jobs`：查询任务列表。
- `GET /api/v1/analysis/jobs/{job_id}`：查询任务详情。
- `POST /api/v1/analysis/jobs/{job_id}/retry`：重试任务。
- `GET /api/v1/analysis/reports/{job_id}`：查询结果。
- `GET /api/v1/analysis/reports/{job_id}/download.json`：下载 JSON。
- `GET /api/v1/analysis/reports/{job_id}/download.pdf`：下载 PDF。

### 10.2 试卷项目 API

当前已有接口方向：

- `POST /api/v1/paper-projects`：创建项目。
- `GET /api/v1/paper-projects`：查询项目列表。
- `GET /api/v1/paper-projects/{project_id}`：查询项目详情。
- `POST /api/v1/paper-projects/{project_id}/files`：上传试卷和标准答案。
- `POST /api/v1/paper-projects/{project_id}/extract`：触发题目提取。
- `POST /api/v1/paper-projects/{project_id}/approve-questions`：审核题目。
- `POST /api/v1/paper-projects/{project_id}/analyze-answer-sheet`：分析单个学生答题卡。
- `POST /api/v1/paper-projects/{project_id}/analyze-students`：批量创建学生分析任务。

### 10.3 掌握度 API

当前已有接口方向：

- `POST /api/v1/mastery/events:ingest`：导入掌握度事件。
- `GET /api/v1/students/{student_id}/mastery`：查询学生掌握度。
- `GET /api/v1/students/{student_id}/report`：查询学生报告。
- `GET /api/v1/exams/{paper_id}/analysis`：查询试卷分析。
- `GET /api/v1/exams/{paper_id}/group-summary`：查询班级知识点汇总。

## 11. 迁移计划

### 11.1 近期迭代

- 固化 v1 主链路：上传试卷、答题卡、标准答案，输出报告和掌握度事件。
- 对 `AnalysisWorkflow` 增加真实阶段，逐步减少 legacy adapter 的职责。
- 先抽离低风险阶段，例如输入校验、结果封装、报告构建、掌握度事件生成。
- 再抽离中风险阶段，例如标准答案解析、对齐判分、错因诊断。
- 最后抽离高风险阶段，例如题目结构识别和学生作答识别。

### 11.2 中期迭代

- 将阶段日志升级为独立表。
- 增加样本集和评测脚本。
- 建立真实模型回归基线。
- 强化试卷项目复用流程。
- 将密钥和敏感配置从 JSON 迁移到环境变量或密钥管理。

### 11.3 后续扩展

- 自动标准答案生成进入可选能力。
- 支持批量学生答题卡。
- 支持人工复核和局部修正。
- 支持多模型 profile 对比。
- 支持知识点图谱维护。
- 支持更完整的班级、年级和考试分析。

### 12.3 真实模型验收

真实模型验收建议使用固定样本集：

- 同一批试卷和答题卡重复运行，比较结果稳定性。
- 对照人工标注，统计题目识别准确率。
- 对照人工批改，统计判分准确率。
- 检查错因诊断是否可被教师复核使用。
- 检查总耗时、失败率和关键阶段耗时。

成功标准：

- 主链路可稳定完成。
- 标准答案判定题数不减少。
- 结果 JSON 字段兼容前端展示。
- 错题诊断和学生画像可用于报告。
- 任务失败时有明确阶段和错误信息。

## 13. 风险与对策

| 风险 | 影响 | 对策 |
| --- | --- | --- |
| 模型输出不稳定 | 判分和诊断波动 | 使用结构化 schema、样本集、重试和置信度 |
| `demo/service.py` 继续膨胀 | 维护成本增加 | 按阶段逐步抽离，不再向大文件堆新能力 |
| 输入模式过多 | 流程复杂、测试困难 | v1 收敛到上传试卷 + 答题卡 + 标准答案 |
| 自动标准答案混入主流程 | 结果可信度不清晰 | v1 主流程只认上传标准答案，自动生成后置 |
| 大模型耗时过长 | 用户等待不可接受 | 分块、并发、按需执行、fast profile |
| 密钥配置泄漏 | 安全风险 | 密钥环境变量化，禁止提交真实 key |
| 缺少评测集 | 优化不可量化 | 建立固定样本和指标基线 |

## 14. 设计原则落地

- KISS：v1 只保留最稳定主链路，避免把 demo 所有模式一次性平台化。
- YAGNI：自动标准答案、批量批改、多模型对比等能力后置。
- SRP：API、应用服务、领域模型、基础设施、分析阶段各自承担单一职责。
- OCP：VLM/LLM、YOLO、PDF 转图、报告生成通过适配器扩展，不让业务流程绑定具体实现。
- DIP：应用层依赖端口协议和抽象能力，而不是直接依赖具体模型供应商。
- DRY：文件保存、任务状态、阶段日志、报告导出等公共能力沉淀为服务或基础设施模块。
